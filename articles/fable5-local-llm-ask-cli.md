## 完成物より、「仕事の進め方」そのものに驚いた話

2026年7月2日早朝、日課のワークアウトを終えてClaude Codeを立ち上げたらfable5が使えるようになっていた。

この記事は、その実装会話ログを時系列で再構成したものである。

> セキュリティ上公開できない内部パス、IPアドレス、顧客固有情報、無関係な作業断片は削除または一般化しています。
> ただし、判断の順序、途中の失敗、未検証事項、設計変更の経緯は、都合よく整え直していません。
> なお、本文中「引用する」と断って示すfable5の発言ブロックは、実際のセッションログからの原文引用です（［　］は固有名を一般化した箇所。それ以外は文言・強調とも原文ママ）。

「fable5が一発で全部作りました」というfable5スゲー＆成功談ではありません。

fable5級のAIと一緒に実装を進めるとき、何が本当に価値になるのかを、自分自身が見せつけられた…そのありのままの記録。

自分にとってfable5の使い方が一番しっくり来たので記事にした次第。では始めます。

## はじめに：これは小さなCLIツールを作ったという話の披露ではない

最終成果物だけを見れば、たいしたものではない。

ターミナルでこう打つ。

```bash
ask "7日以内に更新されたファイルを探したい"
```

すると、LAN内で稼働しているローカルLLMが、たとえばこう返してくる。

```bash
find . -type f -mtime -7
```

それだけだ。

Ubuntuならbash版、WindowsならPowerShell 7版がある。
質問文をローカルOllama APIに投げ、回答をそのままターミナルへ返すだけの、非常に小さなCLIである。

名前は `ask` にした。

```bash
ask "圧縮したtar.gzを展開したい"
ask "ポート8080を使っているプロセスを特定したい"
ask "サービスのログを直近だけ見たい"
```

こういう、「知っているはずなのに毎回うろ覚えになる」シェルコマンドを、日本語で聞く。

一見すると、単なる自然言語コマンド生成ツールに見えると思う。

でも、俺にとって面白かったのはCLIそのものではない。

このツールを作るまでの過程で、Claude Codeのfable5が、

* 最初に提示した解決策の前提そのものを疑い、
* 既存プロジェクトと実機環境を自分で読み、
* 必要な調査を並列化し、
* モデルを雰囲気で決めず評価器を作り、
* 物理GPU構成に起因する既知の罠まで避け、
* 実装し、
* エラー系を検証し、
* 未検証のPowerShell部分は未検証のまま明記し、
* README、利用ガイド、引き継ぎ記録、Gitコミットまで閉じた。

そして途中で、ローカルLLMが失敗した実例を見つけると、すぐに「モデルが小さいからなのか」「推論を切っているからなのか」を切り分ける実験まで設計した。

この一連の仕事の進め方が、すごかった。

なので、この記事はツールの実装記事ではなく、fable5がどのような推論をしたか、その結果としてどんな回答をして、タスクを自律的にこなしていったか？それを伝える記事である。

コードを書くこと自体ではない。
「何を調べるべきか」「何を測るべきか」「どこは未確定なのか」「どこからは人間の確認が必要なのか」を、かなりまともに仕事として扱っていた。

この記事は、その実録記事である。

## 1. そもそもの悩み：シェルの知識はある。でも毎回、脳内検索が面倒

昔から、シェルコマンドの本は何冊も買ってきた（笑）

`find`、`grep`、`awk`、`sed`、`xargs`、`lsof`、`journalctl`、`systemctl`、`tar`、`rsync`、`ss`、`ip`、PowerShellの `Get-*` 系。

全部、一度は使ったことがある。
大体の方向性も分かる。

でも、現場で必要になるのはいつも細部だ。

* `find` の「3日以内」は `-mtime -3` だったか、`+3` だったか
* `tar.gz` の展開は `xzf` だったか、どこに `-C` を置くんだったか
* あるポートを掴んでいるPIDを出すには `lsof` のどのオプションか
* PIDから実行ファイル実体を見るなら `/proc/<PID>/exe` だったか
* systemdで失敗理由を見る時、直近だけ絞るにはどうするか
* PowerShellでサービス、イベントログ、ネットワーク、プロキシの実効設定を見るには何を組み合わせるか

「検索すれば出るじゃん？」はその通り。
ただ、毎回検索して、似た回答をいくつか見て、今のOSやシェルに合うものを選ぶ。

その数十秒、数分の小さな摩擦が、一日中積み重なる。

業務の中に、閉域網で運用される業務システムの保守がある。

扱う情報の性質上、インターネットに何も出せない種類のネットワークだ。

ユーザー先に設置された端末を、リモートで調査することがある。向こうはWindows、sshで乗り込み、PowerShellを使う。

現場で「あのコマンド何だったっけ」が発生した時、わざわざ別端末で調べるのは地味に面倒なのだ。

だから最初に考えたのは、こういうことだった。

> シェルからローカルLLMに聞けたら楽じゃないか？

クラウドAIに貼り付けるのではない。
自分のLAN内にあるGPUサーバへ投げる。

LinuxでもWindowsでも、ターミナルから一発。

```bash
ask "ディスクの空き容量を見たい"
```

```powershell
ask "直近30分のイベントログからエラーだけ見たい"
```

これで十分な答えが返ってきたら、コマンド辞典を探す回数はかなり減る。

少なくとも、俺自身が楽になる（笑）

## 2. 最初の依頼は「Claude Code経由でローカルLLMを使えるか」だった

最初から `ask.sh` 、つまりCLIツールを作るつもりで始めたわけではない。

俺がfable5に最初に聞いたのは、もっと技術的な話だった。

というのも、Claude Codeは接続先を替えることで、ローカルLLMを使えるはずで、側（「ガワ」）としてClaude Codeを使うつもりだった。

* ローカルにはOllama APIで叩けるLLMがある(3GPU)
* 別系統でOpenAI互換APIのローカルLLMもある(LMStudio 1GPU)
* Claude Codeには特定モデルや接続先を変える仕組みがあるはず
* これらをどう組み合わせたら、UbuntuやWindows PowerShellで「何のコマンドを打てばいい？」に答える仕組みを作れるか

このように前提を伝えた場合、普通なら、ここでClaude Codeの設定方法を調べて終わる。

`ANTHROPIC_BASE_URL` がどうとか、ローカルAPI互換がどうとか、LiteLLMを挟むかどうか、という話になる。

それも間違いではない。というか、むしろ、そういう回答に慣れてしまっている自分が居る。
実際、fable5はその調査もした。

ただ、最初の数分でfable5は、こちらの質問に直接答える前に、こういう方向へ話を持っていった。実際の回答から引用する。

> この時点で一つ重要な論点が見えています。**「完全ローカル＋一発呼び出し」だとClaude Codeを経由すること自体の妥当性が問われます**。Claude Codeはシステムプロンプトが巨大（数万トークン規模）で、小型ローカルモデルにはコンテキスト圧迫・レイテンシ・ツール呼び出しプロトコル対応の三重苦になる可能性があります。

これが最初の「おっ」と思った点だった。

俺は「Claude CodeとローカルLLMをどうつなぐか」を聞いていた。
でもfable5は、「つなげる方法」より先に、「そもそもつなげる必要があるか」を疑った。

Claude Codeを経由する場合、便利なものも多い。

* 既存のCLI UX
* ツール呼び出し
* 設定ファイル
* モデル切替
* フックやMCPとの接続
* エージェント的な作業能力

ただし、今回必要なのは「コマンドを一つ提案してほしい」だけだ。

そこへ巨大なシステムプロンプト、ツール定義、エージェント用の文脈、モデル互換性、Anthropic形式のtool use、コンテキスト消費、起動コストまで持ち込む必要があるのか。

小型ローカルLLMにとっては、むしろ不利になる可能性がある。

つまり、問題は「Claude CodeでローカルLLMを使う方法」ではなくなった。

> ローカルLLMへ、最小限の目的に最小限の形で聞くには、何を捨てるべきか。

この時点で、実装の方向はかなり変わった。

## 3. fable5が最初にやったのは、コードを書くことではなく、仕事を分割することだった

この種の話でありがちな失敗は、いきなり実装に入ることである。

適当にPythonかBashでHTTPを叩き、ローカルLLMから返ってきた文章を表示する。
それだけなら30分でできる。

でも、それで「現場で使えるもの」になるかは別問題だ。

fable5はまず、既存情報を三方向から並列に調べた。

### 3.1 既存のローカルLLM評価基盤を読む

俺の手元にはすでに、ローカルLLMの実務適用性を測るための評価プロジェクトがあった。

そこには、

* どのモデルが動いているか
* どのGPUでどのモデルが安定するか
* Ollamaのポート構成
* thinkingを切ると何が起きるか
* context windowが小さいとどう壊れるか
* 複数GPUで特定モデルがどう壊れるか

といった、単なる「モデル一覧」ではない実運用上の知見が残っていた。

今回のようなCLIを作るなら、それを活用するのは当然のながれ。

### 3.2 既存のエージェント基盤を読む

また、俺の別プロジェクトには、複数エージェントをRoom単位で動かし、永続化や監査、コンテキスト隔離まで持つ基盤があった。

技術的には、そこへOllama adapterを足すことは難しくない。

実際、既存設計はadapter抽象化がされていて、ローカルLLMを足す余地もあった。

しかしfable5は、「足せる」と「足すべき」を分けた。

今回欲しいのは、複数のAI社員が非同期に議論する仕組みではない。

```text
ask "この症状を確認するにはどのPowerShellコマンドを打てばいい？"
```

という、一問一答の道具である。

そこへRoom、SQLite、監査ログ、役割分担、コンテキスト隔離まで持ってくるのは、明らかに重い。

技術的に統合可能であることを確認した上で、**統合しない**という判断が出た。

これは実装上、とても大事である。

できるから入れる、ではない。
今回の目的に対して余計なものは入れない。

### 3.3 Claude CodeとローカルLLM接続の仕様を調べる

並行して、fable5はClaude CodeをローカルLLMへ接続する方法もちゃんと調べていた。

結論だけ言えば、今の環境なら接続自体は可能である。

Ollama側のAnthropic互換エンドポイント、各種環境変数、ローカルゲートウェイ、OpenAI互換APIの変換レイヤーなど、選択肢はいくつかある。

ただ、今回の結論は「Claude Code経由はできる。しかし、今回の一発Q&Aに対しては、直にOllama native APIを叩く方が小さく、速く、壊れにくい」となった。

その根拠の説明を引用する。

> Claude Codeのシステムプロンプトは約12Kトークンで、ローカルエンドポイントでは**prompt cachingが効かない**（毎回フル再処理）。RTX 3060で7〜9B Q4のprefillは数百tok/s程度なので、**1文字も出力される前に十数秒〜数十秒**かかる計算になります。一方［既存の評価プロジェクト］の実測では、think=off＋短いプロンプトなら**1コール中央値1.3〜2.7秒**。「サクッと聞く」UXの成立条件はプロンプトを小さく保つことそのもので、これはClaude Codeという側の構造と真っ向から衝突します。`--bare` を使っても12K本体は消えません。

推測で「重そうだからやめよう」ではない。
**トークン数、prefill速度、実測レイテンシという検証可能な数字で、「なぜ経由しないのか」が説明**されている。

これで、アーキテクチャはほぼ決まった。

```text
[Ubuntu bash / Windows PowerShell]
                ↓
             ask
                ↓
      Ollama native API /api/chat
                ↓
       LAN内のローカルGPUサーバ
                ↓
            ローカルLLM
```

クラウドは介在しない。
Claude Codeも、実行時の経路には入らない。

Claude Codeは、この仕組みを考え、測り、作り、検証する側に回る。

この役割分担が、かなり気持ちよかった。

## 4. モデルを「何となく」で選ばなかった

ローカルLLMの話になると、すぐにこういう会話になりがちである。

> 何Bのモデルがいいですか？
> 最近出たモデルは強いですか？
> ベンチマークで何点ですか？

もちろん重要だ。

でも、今回必要なのは、一般的な知能テストではない。

必要なのは、

> 日本語で聞かれたLinux bashまたはPowerShellの質問に対して、現場で使えるコマンドを返せるか。

である。

例えば、以下のような問題だ。

* 直近7日以内に更新されたファイルを探したい
* ポートを使っているプロセスを特定したい
* サービスログを直近だけ見たい
* 圧縮ファイルを展開したい
* 特定のPIDの実行ファイルを知りたい
* Windowsでイベントログを絞りたい
* PowerShell 7でファイルを再帰検索したい
* 破壊的な操作をする前に安全な確認コマンドを出してほしい

fable5は、ここを配管の問題ではなく、測定の問題として扱った。当時の回答を引用する。

> **一番大事な未解決点: 「精度良く」の裏付けがまだゼロ**
>
> ［既存の評価プロジェクト］の100ケースは全部**門番（根拠強制の弁別）タスク**で、コマンド生成・シェル指南は一件も測っていません。qwen3-8bが門番で良かったからといってコマンドQAで良い保証はない——これはあなた自身のプロジェクトが立てた原則（転移を仮定しない）です。つまりゴール達成のレバーは配管よりも、**コマンドQA用の小さなevalケース（Ubuntu/PowerShell各25問程度、検証可能な正解つき）を作って手持ちモデルをスイープし、チャンピオンを決める**ことにあります。

「あなた自身のプロジェクトが立てた原則」——別の評価プロジェクトで俺自身が確認していた **「あるタスクの成績を別のタスクに転用しない」** という原則を、こちらへ突き返してきたわけだ。

こうして、専用の評価基盤が作られた。

bash 25問、PowerShell 25問。合計50問。

複数のローカルモデルに対して、同じ質問を投げる。
応答全文、レイテンシ、エラー、判定結果を保存する。
自動判定で怪しいものは、別のjudgeに回して実際に検証する。

ここで重要なのは、単に正答率を見るだけではない。

* コマンドとして構文が通るか
* 要件を満たしているか
* 破壊的操作に対して危険な答えを出していないか
* bashとPowerShellで環境非互換のコマンドを出していないか
* 説明はもっともらしいが、コマンド自体は間違っていないか
* `head -n` の行数のような、実行すると初めて分かる微妙なズレがないか

まで見る。

これが大事だった。

LLMの回答は、文章としてはもっともらしい。
でも、現場で欲しいのは「もっともらしい説明」ではない。

**打てるコマンド**である。

## 5. 実機の癖まで織り込んで、評価を回す

この評価で、fable5の動きがもう一段面白かった。

俺のGPUサーバは、複数枚のRTX 3060 12GBを載せたUbuntu機である。
モデルによっては複数GPUにまたがってロードできる。
ただし、モデルによっては複数GPU splitで不安定になる。

特定のGemma系モデルには、すでに「複数GPU splitに載せると壊れやすい」という既知の罠があった。

普通なら「7モデルを並列評価します」で終わるところを、fable5は既存ログを読んでいたので、評価の割り当てを変えた。

* 複数GPUで安定して動くモデル群は、2GPU側のOllamaへ
* 問題のあるモデルは、単独GPU側の別Ollamaへ
* モデル切替時にはVRAMを解放
* qwen系はthinkを切る
* contextは明示する
* モデルロード込みの時間とウォーム時の応答時間を分ける
* 全回答を後から検証できる形で残す

つまり、「ベンチマークを回す」というより、**この機械で、この条件なら比較できるという実験環境を組んだ**。

途中では集計処理に、結果を汚染し得るバグも見つかった。

fable5はそれを自分で見つけ、修正してからフルスイープに進んだ。

こういうところが、俺はすごいと思う。

最終的に正しい表だけ出てきても、その表が途中で壊れていたかもしれない。
でもログには、「あ、ここ集計がおかしい」と自己発見して、直した痕跡が残っている。

結果だけ見ると地味だ。
でも、実務ではこっちの方が大事だったりする。

## 6. 結果：標準モデルはqwen3:14b、ただし万能ではない

50問のシェルコマンド評価を回した結果、標準モデルとして選ばれたのは `qwen3:14b` だった。

実測は、正答率92%、ウォーム時の応答中央値2.7秒。

ただし、選定理由は単純な最高点争いではない。

* 標準モデルとして十分な精度
* ウォーム時の応答が数秒
* コマンドQ&Aとして使うには待ち時間が現実的
* 危険な操作で明確な安全違反を出していない
* 実際の質問に対する回答の質と速度のバランスがよい

という、実務上のバランスである。

実際、評価の途中でfable5が見ていたのは、点数そのものより「失敗の質」だった。judge判定の途中経過に対するコメントを引用する。

> **qwen3.5:4bは捏造が出る**: 実在しないコマンドレット（`New-MsSchedule`）を作り出すなど、小型化の代償が判定不能ではなく「もっともらしい嘘」として出る——ask用途では一番危険なタイプ

そして、本命モデルにも容赦がない。

> 注目点: **本命候補のqwen3:14bにも捏造が1件出ました**（`Get-EnvironmentVariable`——存在しないコマンドレット）。無傷のモデルはいない構図です。あとPowerShellの`Get-EventLog`（PS7非対応）は複数モデルが繰り返し踏んでおり、「学習データが古い5.1時代の知識で答える」という小型モデル共通の弱点がはっきり見えてきました。

「どのモデルが一番賢いか」ではなく、「どのモデルの失敗なら現場で許容できるか」を見ている。
選ばれたqwen3:14bも、無傷だから選ばれたのではない。

結果として、基本形はこうなった。

```bash
ask "7日以内に更新されたファイルを探したい"
```

標準は `qwen3:14b`。
thinkは切る。

ウォーム状態なら、体感で数秒。

さらに、単純で急ぎの質問用に軽量モデルを逃がす。

```bash
ask --fast "プロセスをメモリ使用量順に見たい"
```

これは軽量モデルへ送る。

ここで大事なのは、ローカルLLMを「一つの賢さ」で扱っていないことだ。

* 普通の質問には標準モデル
* 急ぎの単純質問には軽量モデル
* 必要なら一時的にモデル指定
* 破壊的操作は人間が裏取り
* そもそも実行は自動化しない

このように、役割を分けた。

CLIのヘルプはこんな感じになった。

```text
使い方: ask [options] "質問文..."

例:
  ask "7日以内に更新されたファイルを探したい"
  ask --fast "プロセスをメモリ使用量順に見たい"
  ask --model qwen2.5:7b "圧縮したtarを展開したい"

options:
  --model <name>   このコマンドに限りモデルを一時上書き
  --fast           軽量・低精度フォールバック
  --quiet          末尾の注意書きを出さない
  -h, --help       このヘルプを表示
```

返答の末尾には、必ずこれを付けた。

```text
⚠ local LLM: 実行前に --help 等で裏取り推奨
```

これは飾りではない。

評価の実測が92%ということは、残り8%は間違うということだ。

92%は便利だ。
でも、100%ではない。

だから `ask` は「コマンドを実行するエージェント」ではなく、**人間へ実行候補を返す辞典**として設計した。

ここはかなり意識している。

## 7. 「便利」と「勝手に実行する」は別物

AIにシェルを触らせる話になると、すぐに「エージェントに自動実行させる」という方向へ行きやすい。

でも、保守現場で本当に欲しいのは、必ずしも全自動でやってくれることではない。

現状のLLMの性能とハーネスでそれはむしろ怖いし、やってはいけない系だとさえ思っている（笑）。

* 消すべきでないものを消す
* 本番サーバで間違ったサービスを止める
* 想定外のファイルを上書きする
* 権限を変える
* Windowsの設定を壊す
* ユーザー環境の調査ログを勝手に外部送信する

こういう事故は、AIが「便利そう」に自律実行することで起きる。

だから今回の `ask` は、最初から実行しない。

質問を投げる。
コマンド候補と説明が返る。
最終的に打つのは人間。

破壊的な操作なら、`--help`、`man`、`Get-Help`、dry-run、対象一覧の確認を挟む。

それでいい。

というか、現場ではその方がいい（ここ皆さんいろんな意見があるとは思いますが）。

全自動にさせたいなら、それは決定論的にプログラムで組むべき。「判断」が介入する必要があるなら、そこには常に「？」をつけて検討すべき。
エスカレーションさせるのか、精緻な分類器をLLMに任せるのか、これは自分の持論ではあるが。

AIの価値は、勝手に触ることではない。
人間が知っているが、毎回思い出すのが面倒な構文や手順を、必要な瞬間に引き出してくれることだ。

## 8. Bashだけで終わらせず、PowerShellまで持っていった

俺の現場にはLinuxだけでなくWindowsがある。

しかも、単なるWindowsではない。
リモート保守の現場で、ユーザー先端末にSSHで入り、PowerShellを使う。

だからbash版だけ作って「完成」とは言えない。

fable5は、Ubuntu bash版とWindows PowerShell 7版の2クライアントを作った。

PowerShellは、特に罠が多い。

例えば、プロファイルに関数を登録する場合、こう書かなければならない。

```powershell
function ask {
    & "C:\tools\ask\ask.ps1" @args
}
```

この2文字には意味がある。

```powershell
&
```

これは呼び出し演算子だ。

ドットソースではない。

```powershell
. .\ask.ps1
```

のように読み込むと、スクリプト内部で `exit` が呼ばれた時、対話中のPowerShellごと閉じる可能性がある。

そして引数転送も、こうでなければならない。

```powershell
@args
```

単に `$args` を渡すと、場合によっては配列が一個の文字列に潰れて壊れる。

こういうのは、普段PowerShellを触っていても普通に踏む。

さらに、PowerShellのJSON化はデフォルト深さが浅い。

APIペイロードを `ConvertTo-Json` する時、ネストした `messages` や `options` が文字列に潰れる危険がある。

`$ErrorActionPreference = 'Stop'` と `Write-Error` の組み合わせも、例外処理を意図せず壊し得る。

HTTP例外を型で分けようとしても、PowerShell内部の継承関係で期待どおりに分岐しないことがある。

このへんをfable5は、Windows実機が手元にない段階でセルフレビューし、READMEへ「既知の落とし穴」として残した。

もちろん、未実機検証のものを「動いた」とは言っていない。

Windows版には、初回確認手順が書かれている。

1. PowerShell 7であること
2. 実行ポリシー
3. ダウンロード由来ブロックの解除
4. 日本語文字化け
5. GPUサーバへのAPI疎通
6. 存在しないエンドポイントでの異常系
7. `--fast` と `--model` のモデル切替
8. 全部通って初めて実運用

ここが、俺は大事だと思う。

AIが書いたPowerShellスクリプトは、実機で試していなければ未検証である。

当たり前のことだが、AIを使っていると、この当たり前が崩れやすい。

fable5は少なくとも、未検証部分を未検証として残した。

## 10. 実際に便利だった例：コマンド辞典を頭の外に出せる

```text
findだったっけ？
```

## 11. そして失敗した：パイプライン合成は、thinkを切ったローカルLLMには重かった

ここからが、今回一番面白かったところである。

ある時、こんな質問を投げた。

```bash
ask "lsofでわかるPIDをキーにして、そのプロセス自体の実行ファイルのパスを知りたい。ワンライナーで教えて"
```

返ってきたのは、ぱっと見それっぽい、しかし実際には破綻したパイプだった。

```bash
lsof -t -c $(ps -eo comm,pid | awk '{print $1 " " $2}' | grep -v PID | tr ' ' '\t' | cut -f2) | xargs -I {} readlink /proc/{}/exe
```

これはダメである（笑）

何がダメか。

* `lsof -c` はPID指定ではなく、コマンド名指定
* 「lsofで分かったPIDを使う」という元の要求を途中で忘れている
* なぜか全プロセスを `ps` で列挙し直している
* パイプ全体のデータフローが崩れている
* 最後の `/proc/<PID>/exe` と `readlink` の発想だけは合っている

つまり、知識が全くないわけではない。

`lsof` を知っている。
`ps` を知っている。
`awk` を知っている。
`/proc/<PID>/exe` を知っている。
`readlink` を知っている。

でも、「前段が何を出し、次段が何を受けるか」を最後まで保持しながら、パイプを組み立てられていない。

この失敗を見て、俺はこう言った。

> ああ、こういうコマンドをパイプでつないで処理する話は、推論を切っている分弱いな。
> 今のファーストバージョンには荷が重いタスクだな（笑）

普通なら、ここで正解を教えて終わる。

でもfable5は、そこで終わらなかった。

## 12. 「モデルが小さいから」ではなく、「推論がないから」なのかを測った

俺の愚痴に対するfable5の返答を、そのまま引用する。

> いい失敗サンプルですね（笑）。これはまさにevalで見えていた「多段階・合成タスクの弱さ」の実地再現です——単発コマンド質問は92%取れるが、パイプの組み立ては「構成を考える」作業でthink=offが一番効くところ。ただ「本当に推論の問題なのか、モデルサイズの問題なのか」は実測で切り分けられるので、同じ質問をその場で2条件に投げてみます: **qwen3:14bのthink=ON**と、**寝ているqwen3.6:27b**（think=OFF）です。

愚痴が、その場で対照実験に変わった。

これが本当に良かった。

ローカルLLM界隈では、失敗するとすぐに「もっと大きいモデルを使え」という話になりがちである。

でも、それは本当に正しいのか。

数分後、実測が返ってきた。

> 実測結果が出ました。**あなたの読みどおり「推論がない分弱い」で正解**——そして直し方も特定できました。

結果は、かなり明確だった。

| 条件                      |                  実測 | 出力の評価          |
| ----------------------- | ------------------: | -------------- |
| qwen3:14b / think OFF   |                2〜3秒 | 破綻したパイプ        |
| qwen3:14b / think ON    |               17.5秒 | 正しいパイプ         |
| qwen3.6:27b / think OFF | ウォーム換算で数秒台、初回はロード込み | 惜しいが複数PIDで壊れ得る |

同じ14Bモデルでも、thinkをONにしたら、ちゃんとこういう形に到達した。

```bash
lsof -t | xargs -I {} readlink /proc/{}/exe
```

fable5はこれに「特定ポートなら `lsof -t -i :8080 | xargs -I{} readlink /proc/{}/exe` のように `-t` の前に絞り込みを足す形」と添えてきた。

そのポート絞り込み版を、手元で実際に気になっていたポートに当てはめて打つ。

```bash
lsof -t -i :15991 | xargs -I{} readlink /proc/{}/exe
```

返ってくる。

```text
/usr/bin/remmina
```

まさに欲しかったものだ。

ポート15991を掴んでいるPIDを取る。
そのPIDの `/proc/<PID>/exe` を辿る。
実行ファイルの実体が出る。

この一行を、毎回自分で組み立てなくてよくなる。

これは大きい。

そして実験から見えたことも大きい。fable5の総括を引用する。

> つまり構図はこうです:
>
> - **単発コマンド質問**（evalの主戦場）: think=OFFで92%・2〜3秒 → 今のaskが最適
> - **パイプ合成質問**: 「構成を考える」タスクなので推論が本体。同じ14bでもthink=ONにするだけで正答に化ける。ただし代償は17秒
> - サイズを27bに上げるだけ（think=OFF）では惜しいところまでしか行かない——**足りないのは知識ではなく推論**、という切り分けまで取れました

今回足りなかったのは、単なるコマンド知識ではなかった。
複数のコマンドを、入力と出力の関係を保ちながら組み立てる推論だった。

これは、実装の次の一手をかなり明確にした。

## 13. v2は「巨大モデル化」ではなく、`--think` という二段変速になる

この実験結果を受けたfable5の提案が、これだ。

> なのでv1の守備範囲外という判断は正しくて、かつ**v2の最有力は `--think` フラグ**です（実装はpayloadに`"think": true`を足すだけ、両スクリプト各1行＋ヘルプ数行。「じっくり考えてほしい時だけ17秒払う」という明示的なギアチェンジ）。evalの知見どおりthink=ONは決定論が崩れますが、対話用途では実害なしです。やるならすぐ入れます。

失敗の観測から、切り分け実験、そしてUIの設計変更まで、一本の線でつながっている。

通常は、thinkを切る。

```bash
ask "ポート8080を使っているプロセスを知りたい"
```

これは2〜3秒で返ってきてほしい。

単発のコマンド想起なら、それで十分に便利だ。

でも、パイプラインの合成、複数条件の組み立て、データフローの保持、ちょっと複雑なワンライナーが必要な時だけは、ユーザーが明示的にギアを上げる。

```bash
ask --think "15991番ポートを使っているプロセスの実行ファイルを知りたい"
```

数秒ではなく、十数秒かかるかもしれない。

でも、その時は「考える時間」を払う価値がある。

これは、俺にはかなりしっくり来た。

ローカルLLMに、何でも一律に期待しない。

* 普段は高速なコマンド辞典
* 必要な時だけ、少し待って推論モード
* それでも最後は人間が確認して実行

この二段変速の方が、毎回大きいモデルを載せて待つよりずっと実用的だと思う。

ローカルLLMを万能AIとして扱うのではない。

能力の境界を測り、そこに合わせて道具のUIを作る。

これが今回の一番大きな収穫だった。

## 14. 追試：`--think` は銀の弾丸ではなかった

その後、`--think` を実装して、早速試した。

§13で例に出した、まさにあの聞き方である。

```bash
ask --think "15991番ポートを使っているプロセスの実行ファイルを知りたい"
```

返ってきたのは、これだった。

```bash

```

> このコマンドは、15991番ポートを使用しているプロセスの情報を表示します。出力には、プロセスが関連付けられている実行ファイルのパスも含まれます。

違う。期待した答えはそうじゃない！（笑）
実際に打ってみると、こうなる（USER列は伏せてある）。

```text
COMMAND    PID  USER   FD   TYPE  DEVICE SIZE/OFF NODE NAME
remmina 165512 user    28u  IPv4 5736018      0t0  TCP localhost:56524->localhost:15991 (ESTABLISHED)
```

出るのはプロセス名までで、実行ファイルのパスは出ない。
説明文の「実行ファイルのパスも含まれます」は、もっともらしいが事実ではない。

「フルパス」と言い足しても、結果は同じだった。

```bash
ask --think "15991番ポートを使っているプロセスの実行ファイルのフルパスを知りたい"
```

```bash
lsof -i :15991
```

thinkをONにしているのに、である。

つまり、推論を許可しただけでは、モデルは考え始めない。

```bash
ask --think "15991番ポートを使っているプロセスの実行ファイルの名前だけではなく、その実行ファイルがどこに配置されているのか、フルパスも知りたい"
```

今度は来た。

```bash
lsof -t -i :15991 | xargs -I {} readlink /proc/{}/exe
```

実行する。

```text
/usr/bin/remmina
```

正解である。

`--think` は「考える能力」を解放する。
でも、「何を考えるべきか」は質問文が与える。

「lsof単発では足りない」と分かる程度に要求が明示されて、初めてパイプの合成が始まる。
曖昧な質問は、推論モードでも曖昧なまま処理される。

考えてみれば、これは人間相手でも同じだ。
「実行ファイルを知りたい」と言われたら、プロセス名が分かれば十分だと解釈する人だっている。

ローカルLLMの推論は、行間を埋める魔法ではない。
要求を正確に言語化するのは、変わらず人間の仕事である。

## 16. fable5の「正しい使い方」とは何だったのか

「fable5ってすごいね」という話に見えたかもしれない。

でも、今回うまくいった理由は、最初にある程度の契約を渡していたからだ。

* ゴールは明確にした（PowerShellも7を指定した上でやらせた）
* ローカル完結という制約を渡した
* 既存プロジェクトを読むように言った
* 必要なら質問してよいと言った
* 自律的に動ける部分は動いてよいと言った
* ただし、エージェントの使い分けは難度に応じて考えるように言った
* 処理内容は詳細ログに残すように言った
* 未確定事項は勝手に確定扱いしないようにした

つまり、丸投げではない。

目的、制約、権限、記録の要件を渡した上で、実行計画の立案と進行を任せた。

fable5はその中で、

* 調査を並列に分解し
* 既存基盤を読んで
* 余計な統合を避け
* 評価器を作り
* 実測し
* バグを直し
* 実装し
* エラー系を確認し
* 未検証点を明記し
* 利用者向けと保守者向けの文書を分け
* 最後に引き継ぎとGitコミットまで行った

ここで価値があったのは、「コードが書けた」ことだけではない。

> 次に何をすべきかを、かなり高い確率で自分で決めながら進めた

ことだ。昨日までのOpus 4.8だったら、以下のような感じになってた。

人間が毎回、

```text
次はREADMEを書いて
次はエラー系を見て
次はPowerShell版を作って
次は未検証箇所を整理して
次は引き継ぎ文書を書いて
次はGitにコミットして
```

と指示し続けなくても、作業が閉じる方向へ向かった。

これが、俺にとってのfable5の凄さだった。

ただ、これについてはfable5自身がおもしろいことを言っていた。セッションの最後、俺が「ものすごい進捗だった」と労った時の返答を引用する。

> ありがとう（笑）。ただ、今日の自走が成立した条件の半分はそっちの仕込みですよ——序盤の3問に即答してくれたことで制約が固まり、「Sonnetで回して物足りなければ上位で」という委任基準と「エージェントに詳細ログを吐かせろ」という一言が、12体投げっぱなしでも後から全部検証できる構造を作った。あれがなければ各エージェントの結果を疑うたびに止まってました。

自走は、丸投げからは生まれない。目的と制約と検証手段を先に渡す「契約」から生まれる。
それをfable5の側から言ってきたのが、今回一番の答え合わせだった気がしている。

## 17. ただし、AIが優秀だからこそ、失敗を消してはいけない

この記事で、わざわざ壊れた `lsof` のパイプを載せたのには理由がある。

AI記事は、成功した出力だけを並べると簡単にすごそうに見える。

でも、それでは実像が分からない。わからないまま実務には持ち込めない。

今回のローカルLLMは、単発コマンドの質問にはかなり強かった。
しかし、パイプライン合成では壊れた。

fable5は、その失敗を隠さなかった。

どういうことか？それは、次の実験の材料にしたんだ。

* think OFFでは壊れる
* 同じモデルのthink ONでは通る
* 大きいモデルでもthink OFFでは惜しい誤答になる
* したがって、知識量だけではなく推論の有無が効いている
* だからUIには `--think` という明示的なギアを持たせる
* そしてthink ONでも、要求が曖昧なら最短の答えに流れる——推論の起動条件は聞き方にある

この流れをfable5自身が検証後、推論結果として出してきたことが重要だと思ったわけ。これがfable5の価値か！って。

AIを使って開発する時、失敗は「失敗」で終わらせるものではない。

能力境界を測るサンプルになる。

これは自分がいつもセミナーで言ってることなんだが、「ハルシネーション」って言う事自体が人間の「幻想」「おごり」だろ？って。

人間関係と同じ、普段「xxさんが言ってたから間違いない！」って自分で裏取りもせず言いがちな人ほど「ハルシネーション」って騒いでる印象がある（笑）

LLMは言われたとおりの処理で確率論的に出力してるだけで、わざわざ嘘をつくようにプログラムされてるわけではない。それ（回答）をどう使うか？だけの問題。

キツイ言い方をすれば、意図がないところに意図語彙を使うから、「騙された」という被害者の構図が生まれて、本来やるべき裏取りの怠慢が免責される。そういう構図でしかないわけ。

つまり、回答がハルシネーションであったとしても、そのサンプルを残しておけば、次の設計判断に使える。

だから、今回のログは俺にとって、単なる会話履歴ではない。

ローカルLLMの能力と適用範囲を知るための、リアルな実験記録なわけ。

## 18. まとめ：完成したのはCLIだが、得られたのは「AIと仕事を進める方法」だった

完成した `ask` は、小さい。

* bash版
* PowerShell 7版
* LAN内のOllama APIへ質問
* 標準モデル
* 軽量モデルへのフォールバック
* 一時的なモデル変更
* 安全注意
* 接続、HTTP、空応答などのエラー処理
* 利用ガイド
* 保守用README

技術的に見れば、派手なものではない。

でも、これが日常に入ると、かなり効く。

```bash
ask "直近3日以内に更新されたMarkdownを探したい"
```

```bash
ask "このポートを使っているプロセスの実行ファイルを知りたい"
```

```powershell
ask "WinHTTPのプロキシ設定を確認したい"
```

```powershell
ask --think "サービスは起動しているのに通信できない。NICの確認とFWの設定確認のコマンド教えて"
```

これだけで、昔買ったワンライナー本やコマンド辞典を、毎回開かなくてもよくなる。

もちろん、完全に不要になるわけではない。

正しさの確認は要る。
`--help` も `man` も `Get-Help` も必要だ。
ローカルLLMは環境固有の事情を知らない。
複雑な合成タスクでは壊れる。
思考を有効にすれば遅くなる。

でも、それでいいと思う。

人間が最終判断を握ったまま、日常の「何だっけ？」を数秒で片付けられる。

そして、その道具を作る過程でfable5が見せたのは、単なるコード生成能力ではなかった。

* 問題の前提を疑う
* 調査を分ける
* 実機制約を読む
* 測定器を作る
* 失敗を実験に変える
* 未検証を未検証として残す
* 次の人間が続けられる形に閉じる

こういう仕事の進め方そのものだった。

正直、完成した `ask` も便利で嬉しい。

でも一番印象に残ったのは、

> AIが「正解を答える」だけではなく、
> 不確実な現実の中で、何を調べ、何を測り、何を作り、どこで止まるべきかを考えながら、
> 仕事を前へ進める相棒に確実になり始めている

という感覚、というか実感だった。

それが生ログを読んだ時に一番強く残った。

そして、あのログをそのまま全部公開することはできない。

でも、成功だけを抜き出して「AIがCLIを作りました」と書くのも、何か違う。

だからこの記事では、完成物だけではなく、途中で迷ったこと、捨てた案、壊れた出力、未検証の部分、そこから次の実験へ進んだ流れを残した。

この流れにこそ、今のfable5を使う意味があるんじゃないか？というのがfable復活初日に改めて使ってみて強烈に印象に残ったので記事に残しておこうと思った次第。

## 付記：公開時にGitHubへ置くもの

リポジトリを公開する時は、少なくとも以下を一緒に置くつもりでいる。

* `ask.sh`
* `ask.ps1`
* 利用者向けの `GETTING_STARTED.md`
* 設計と既知の罠を残した `README.md`
* コマンドQ&A評価ケース
* 評価ランナー
* モデル別の実測結果
* 正答だけでなく、代表的な失敗例
* `think=OFF` と `think=ON` の比較
* 「このツールは生成コマンドを自動実行しない」という明確な安全境界

ローカルLLMを使ったツールは、動くデモだけでは足りない。

何ができるかだけでなく、

> 何ができないのか。
> どこで人間が確認するのか。
> なぜこのモデルなのか。
> なぜこの設計なのか。

そこまで含めて公開して、はじめて使えるものになると思う。

fable5が再び使えるようになって真っ先に寝かせていたプロジェクトに適用してみたが、見事に期待を裏切らない結果だったと思っている。
