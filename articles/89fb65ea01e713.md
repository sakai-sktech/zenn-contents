## 背景：「それ、ただの subprocess のラッパーじゃん」

[README](https://github.com/sakai-sktech/open-rx) の PoC を見た人は分かると思います。  
TypeScript が .NET の sidecar を起動して、プロセスの生殺与奪を握って、stdin/stdout で通信する。

で、ここで必ず言われるやつ。

> 「それ、ただの subprocess のラッパーじゃん」

うん。**その通り（笑）**

でもね、これを "ただのラッパー" って言って切り捨てる人は、たぶん一度も **締切** と **クロスプラットフォーム** と **現場の制約** に挟まれて戦ったことがない。

VS Code も Prisma も Docker Desktop も、みんな "ただラップしてる"。  
**だってそれが一番強いから。**

「純度」より「出荷」。これがプロダクトの現実。

というわけで、**誰がどうやって使ってるか** を調べて、腹落ちできるようにまとめました。

## 🌟 どこにでもある "本番の実例"

### 1) OmniSharp（VS Code の C# 拡張）— これが元祖・参照実装

**何をしてる？**  
VS Code で C# の補完、定義ジャンプ、リファクタリング…あれ全部。

**構成はこう：**

*   **親（UI）：** VS Code（Electron / TypeScript）
*   **子（ロジック）：** OmniSharp-roslyn（.NET 実行ファイル）
*   **通信：** LSP を stdin/stdout（または HTTP）で

**なんでこうする？**  
VS Code は TS/Electron。でも C# の解析は Roslyn（.NET）が本体。  
Roslyn を JS に書き直す？……**無理筋（笑）**  
だから .NET を sidecar で起動して、LSP で会話する。以上。

**要するに：**  
あなたが `class Program` と打って補完が出た瞬間、裏では .NET が動いてる。  
TypeScript がそのプロセスを回してる。これが **参照実装**。

**リンク：**

*   [OmniSharp-roslyn](https://github.com/OmniSharp/omnisharp-roslyn)

### 2) Prisma（Node.js ORM）— TSの顔をしたRustエンジン

**何をしてる？**  
TypeScript-first の ORM。DB は Postgres/MySQL など。

**構成：**

*   **親：** Node.js（TypeScript）
*   **子：** Query Engine（Rust バイナリ）
*   **通信：** JSON-RPC を stdin/stdout で

**なんでこうする？**  
DB のクエリ生成やマイグレーション、実際やると重い。  
Node.js 側だけだと詰まる局面が出る。  
だから Rust に寄せて、TypeScript の DX を守る。

**要するに：**  
「重い処理はネイティブ。司令塔は TypeScript」  
これ、いま一番“勝てる型”。

**リンク：**

*   [prisma-engines](https://github.com/prisma/prisma-engines)

### 3) GitLens / VS Code の Git 統合 — "git を書き直さない" が最適解

**何をしてる？**  
blame、history、diff…VS Code の Git 体験を強化するやつ。

**構成：**

*   **親：** VS Code（TypeScript）
*   **子：** `git`（ネイティブバイナリ）
*   **通信：** spawn して stdio を読む

**なんでこうする？**  
Git は20年近く鍛えられた化け物。  
これを JS で書き直すのは、**悪夢というか事故**。  
だから VS Code は `git` を呼んで解析する。シンプルで正しい。

**要するに：**
動いてるものを壊してまで "統一" しない。
**包むのが正解**な世界がある。

### 4) electron-builder — OSネイティブの道具を "まとめて使える" 顔にする

**何をしてる？**  
Electron アプリのビルド・署名・パッケージングを、macOS/Windows/Linux でやる。

**構成：**

*   **親：** Node.js（TypeScript）
*   **子：** OS固有ツール（`codesign`, `signtool.exe`, `rpm`, `dpkg` …）
*   **通信：** CLI を `child_process` で実行

**なんでこうする？**  
署名やパッケージングは OS が握ってる。  
ネイティブツール以外に正攻法がない。  
だから、TS で “統一API” に見せかけて裏で叩く。これが現実の勝ち筋。

**リンク：**

*   [electron-builder](https://github.com/electron-userland/electron-builder)

### 5) GitHub Copilot — "重い処理は別プロセス" がAI時代の標準になりつつある

ここは言い切りすぎると危ないので、**責任ある言い方で**書く。

**何をしてる？**  
VS Code の AI 補完。

**言えること（観測ベース）：**

*   VS Code 拡張は TypeScript
*   Copilot 使用時、環境によっては **追加プロセスが見える**ことがある
*   内部プロトコルや実装は非公開

**要するに：**  
LLM みたいな重い処理を UI プロセスに全部抱え込ませるのは不利。  
だから “UI/制御” と “重い処理” を分ける設計は、AIツールでは自然に増えていく。

**近いOSSの例（ここは断定できる）：**

*   [Tabby](https://github.com/TabbyML/tabby)
*   [Fauxpilot](https://github.com/fauxpilot/fauxpilot)

**参考（高レベル解説）：**

*   [How GitHub Copilot works](https://github.blog/2023-05-17-inside-github-how-does-copilot-work/)

### 6) Docker Desktop — ElectronがDocker Engineを包んでるだけ（そしてそれが最強）

**何をしてる？**  
コンテナ管理GUI。

**構成：**

*   **親：** Electron（TypeScript/React）
*   **子：** Docker Engine（Go）
*   **通信：** Docker API（Unix socket / Windows named pipe）

**なんでこうする？**  
Docker Engine は巨大で複雑。JSで書き直す？無理。  
だから UI は “ただ包む”。それでいい。  
むしろ、それが正しい。

**リンク：**

*   [Docker Engine API](https://docs.docker.com/engine/api/)

## 🎯 いつ使うべきか（"俺の現場"に当てはめるチェックリスト）

このパターンが刺さるのはこういう時：

✅ **既存のネイティブ資産**（C# / Rust / Go / C++）を捨てたくない（捨てられない）  
✅ **性能が要る**（CPU/GPUが重い処理）  
✅ **OS固有APIが要る**（Windows暗号、スマートカード、USB/ドライバ系）
✅ **落ちても巻き込まない**（プロセス分離で安全に）
✅ **LSP**（エディタ拡張は元々この世界観でできてる）

## 🤔「結局ラッパーじゃん」問題に正面から答える

**はい。ラッパーです。だから勝つ。**

よくある批判：

> 「subprocess spawn はハックだ、アーキテクチャじゃない」

……いや、現場で必要なのは **"動くこと"** です（笑）

### 現実チェック

あなたの状況はこれ：

*   ✅ 動く資産がもうある（署名ライブラリ、パーサ、ドライバ、何でも）
*   ✅ 今のプロジェクトは TypeScript
*   ❌ TSに書き直したら数ヶ月。しかもバグる。暗号なら致命傷。

で、選択肢は3つ。

### 選択肢1：全部TSで書き直す

*   **メリット：** “純血”になる（気分はいい）
*   **デメリット：**
    *   時間が溶ける
    *   バグる（暗号は地雷原）
    *   性能が落ちる
    *   二重実装の保守で死ぬ

### 選択肢2：Native Addon（N-API / node-gyp）

*   **メリット：** 低遅延で速い
*   **デメリット：**
    *   Node更新で壊れる
    *   落ちたら親ごと死ぬ
    *   クロスビルド地獄（ARM64/macOS…泣ける）

### 選択肢3：sidecarとして包む

*   **メリット：**
    *   **出荷が速い**
    *   **落ちても巻き込まない**
    *   **デバッグがラク**（stdioが最強）
    *   **クロスプラットフォームに寄せやすい**
*   **デメリット：**
    *   IPCのオーバーヘッド（多くの用途で誤差）
    *   美しくない？…でも**勝つ**

### 結論

「金曜までに出せ」と言われたら、選択肢3が勝ち。  
これはハックじゃない。**実務的なエンジニアリング**。

VS Code は `git` を包んでる。Docker Desktop はデーモンを包んでる。Prisma は Rust を包んでる。
ラップは "恥" じゃない。**勝ち方**です。

## 🚫 逆に、やめとけ（ここ大事）

❌ ただの REST 呼び出しなら `fetch()` でいい  
❌ 単純なロジックなら TS のままが保守しやすい  
❌ 超低遅延を何万回/秒で叩くなら IPC が効いてくる
❌ .NET/Rust を保守できるチームじゃないなら、運用負債になる

## 🛠️ よくある用途（現場で起きるやつ）

### 1) 医療・コンプライアンス

*   **例：** HPKI スマートカード署名（日本）
*   **理由：** 署名資産が .NET 前提、UIはクロスプラットフォームが良い
*   **このrepo：** 出自はここ（[README](README.md)）

### 2) 重い計算のデスクトップアプリ

*   動画、3D、画像処理。JSだけじゃ勝てない領域。

### 3) DB系

*   パース、最適化、マイグレーション。コンパイル済みが強い（Prisma）。

### 4) ハードウェア連携

*   USB/BLE/シリアル。Native Addonよりプロセス分離の方が壊れにくい。

### 5) レガシー統合

*   20年モノの .NET Framework ライブラリ。書き換えるより包め。

## 💡 アーキテクチャ比較（冷静に）

| パターン | 通信 | 良い点 | つらい点 | 向いてる用途 |
| --- | --- | --- | --- | --- |
| **Sidecar（このrepo）** | stdin/stdout | シンプル、デバッグ最強、隔離できる | 型付き契約は自前 | 短〜中寿命タスク / ツール系 |
| **gRPC** | HTTP/2 | ストリーミング、型、生成コード | 仕組みが重い | 長期稼働サービス |
| **Native Addon** | FFI | 最低遅延 | 壊れやすい、落ちると巻き込む | 超ホットパス |
| **REST API** | HTTP | 言語非依存 | 認証/運用が乗る | 分散マイクロサービス |

> 補足：stdin/stdout でも双方向・ストリーミングはできる。gRPCの旨味は"周辺機能（型/生成/エコシステム）"にある。

## 筆者について

CORBA/SOAP 時代から生き延びてるエンジニアです。

2000年代前半に大規模電子カルテ開発で「IDL地獄」「WSDL地獄」を体験し、REST登場時に「やっと来たか！」と感動した世代です。

現在は医療系システム（電子処方箋）を開発していて、「.NET の暗号ライブラリを Linux で使いたい」という課題に対してこのサイドカーパターンに辿り着きました。

技術選定の基準は「締切に間に合うか」「運用で死なないか」「チームが保守できるか」です。理想論より現実主義。
